# QtNixMon 🐧

**Учебный пет-проект: Системный монитор ресурсов для Linux на C++26 и Qt**

> ⚠️ **Важно:** Этот проект создан в образовательных целях (pet project). Он не предназначен для использования в критических производственных системах. Код может содержать баги, архитектура может меняться, а стандарт C++26 находится на стадии разработки.

## 1. Проектирование

### Тема
Разработка высокопроизводительного десктопного приложения для мониторинга системных ресурсов Linux (CPU, RAM) с использованием современных стандартов C++ и фреймворка Qt. Проект служит полигоном для изучения C++26, Qt6 и работы с системными вызовами Linux.

### Что нужно сделать и как
Необходимо создать приложение, которое в реальном времени опрашивает системные файлы `/proc`, обрабатывает данные с использованием современных возможностей C++ (concepts, ranges, modules) и отображает их в виде виджетов Qt. Конфигурация должна сохраняться в JSON.

### Список классов (поля, методы)

1.  **`MainWindow` (UI Layer)**
    *   *Поля:* `Ui::MainWindow *ui`, `QTimer *updateTimer`, `SystemMonitor *monitor`, `ConfigManager *config`.
    *   *Методы:* `constructor`, `destructor`, `onDataUpdated()`, `applyTheme()`, `closeEvent()`.
    *   *Ответственность:* Отрисовка интерфейса, обработка событий окна, связь сигналов/слотов.

2.  **`SystemMonitor` (Backend Logic)**
    *   *Поля:* `std::chrono::milliseconds interval`, `CpuState prevCpu`, `MemoryState prevMem`.
    *   *Методы:* `getCpuLoad()`, `getMemoryUsage()`, `getDiskIO()`, `start()`, `stop()`.
    *   *Ответственность:* Чтение `/proc/stat`, `/proc/meminfo`, вычисления, эмиссия сигналов с данными.
    *   *Технологии:* C++26 Ranges для парсинга, `std::expected` для обработки ошибок.

3.  **`ConfigManager` (Data Persistence)**
    *   *Поля:* `std::filesystem::path configPath`, `nlohmann::json configData`.
    *   *Методы:* `load()`, `save()`, `get<T>(key)`, `set(key, value)`.
    *   *Ответственность:* Сериализация/десериализация JSON, валидация схемы конфига.

4.  **`ThemeProvider` (Styling)**
    *   *Поля:* `QMap<QString, QString> styles`.
    *   *Методы:* `loadStyleSheet()`, `apply(QWidget*)`.
    *   *Ответственность:* Управление QSS стилями (Dark/Light).

### Использование контейнеров для хранения данных
*   **`struct` (aggregates):** Для передачи снепшотов данных (`CpuSnapshot`, `RamSnapshot`).
*   **`std::vector` / `QVector`:** Для списков процессов (в будущих версиях).
*   **`std::map` / `QMap`:** Для хранения настроек конфигурации.
*   **`nlohmann::json`:** Для работы с файлом конфигурации.
*   **`std::optional` / `std::expected`:** Для возврата значений, которые могут отсутствовать или содержать ошибку.

### Сохранение данных (в файл обязательно)
*   **Сколько файлов:** 1 основной файл конфигурации `config.json`.
*   **Путь:** `~/.config/QtNixMon/config.json` (следует стандарту XDG).
*   **Вид хранения:** JSON.
*   **Что хранится:**
    *   `update_interval_ms` (int): Интервал обновления (например, 1000).
    *   `theme` (string): "dark", "light", "system".
    *   `widgets_visibility` (object): `{ "cpu": true, "ram": true }`.
    *   `window_geometry` (array): `[x, y, w, h]`.
*   **Когда:**
    1. При первом запуске (создание дефолтного).
    2. При изменении настроек в UI (автосохранение).
    3. При корректном закрытии приложения.

## 2. Реализация

### План работ (Приоритеты)
1.  ✅**Первое:** Настройка окружения (CMake, Qt6, Compiler с поддержкой C++23/26), создание базового окна.
2.  ✅**Второе:** Реализация `SystemMonitor` (чтение `/proc`), вывод данных в консоль (логирование).
3.  **Третье:** Интеграция данных в UI (QProgressBar, QChart), настройка `QTimer`.
4.  **Четвертое:** Реализация `ConfigManager` (чтение/запись JSON, обработка путей).
5.  **Пятое:** Стилизация (QSS), обработка ошибок, упаковка в AppImage/Deb.

### Модульность
*   **Backend (`SystemMonitor`)** не зависит от Qt (кроме сигналов), может быть протестирован отдельно.
*   **UI (`MainWindow`)** только отображает данные, не содержит бизнес-логики вычислений.
*   **Config** вынесен в отдельный модуль для легкого расширения.

### Валидация (обработка правильности данных)
*   **Файлы:** Проверка существования `/proc/stat`. Если нет — критическая ошибка.
*   **JSON:** При загрузке проверяется схема. Если поле отсутствует — используется дефолтное значение. Если файл битый — создается новый.
*   **Диапазоны:** Проверка значений CPU/RAM (0-100%). Если >100% (баг парсинга) — ограничить до 100%.

### Поиск / Фильтрация / Сортировка
*   **Текущая версия (MVP):** Агрегированные данные (общая загрузка).
*   **План (v1.1):** Вкладка "Processes".
    *   Сортировка по CPU/RAM (использование `std::ranges::sort`).
    *   Фильтрация по имени процесса.
    *   Поиск по PID.

### Обработка ошибок
*   **Исключения:** Использование `std::expected` вместо исключений для производительности.
*   **UI:** При ошибке чтения данных показывать "N/A" или красный индикатор, не крашить приложение.
*   **Config:** Если нет прав на запись в `~/.config`, предупредить пользователя (лог в stderr или ошибка в gui).

### Как запустить
```bash
# Клонируйте репозиторий
git clone https://github.com/1-Aroil-1/QtNixMon.git
cd QtNixMon

# Создайте директорию сборки
mkdir build && cd build

# Конфигурация (убедитесь, что Qt6 найден)
cmake .. -DCMAKE_BUILD_TYPE=Release

# Сборка
make -j$(nproc)

# Запуск
./QtNixMon

# Запуск с GUI
./QtNixMon --gui
```
